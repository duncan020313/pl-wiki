# 코딩 에이전트: 코드베이스를 스스로 읽고 수정하며 검증하는 AI 개발 도구

## 1. 개요

코딩 에이전트는 언어 모델(large language model)의 언어 이해 능력을 기반으로,
코드베이스를 직접 읽고, 터미널을 조작하며 코드를 수정하고, 검증까지 스스로 수행하는 소프트웨어다.
코딩 에이전트의 핵심은 스스로 터미널을 조작하여 기존 도구(`grep`, `find`, 린터, 포맷터 등)를 활용할 수 있다는 점이며,
이를 통해 코드 저장소 전체를 훑어 맥락을 파악하고, 코드 작성부터 검증까지 이어지는 작업 흐름을 처리한다.

## 2. 기존 코드 생성 모델과의 차이

| 구분            | 일반 코드 생성 모델              | 코딩 에이전트                                             |
| --------------- | -------------------------------- | --------------------------------------------------------- |
| 입력            | 짧은 질문, 함수 설명, 코드 조각  | GitHub 이슈, 자연어 요구 사항, 코드베이스 전체            |
| 출력            | 코드 조각, 설명                  | 수정된 파일, 테스트 결과, 패치, Pull Request              |
| 코드베이스 이해 | 사용자가 제공한 일부 코드에 의존 | 파일 탐색, 검색, 코드 읽기를 통해 직접 수집               |
| 행동 방식       | 한 번에 답변 생성                | 관찰, 수정, 검증 반복                                     |
| 검증            | 사용자가 직접 실행               | 에이전트가 테스트나 빌드를 직접 수행할 수 있음            |
| 책임 구조       | 사용자가 코드를 복사해 적용      | 에이전트가 실제 패치를 만들고 변경된 코드를 사용자가 리뷰 |

## 3. 코딩 에이전트의 구성 요소

코딩 에이전트는 (1) 언어 모델, (2) 샌드박스(Sandbox), (3) 에이전트 제어 장치(Agent Harness)가 결합된 소프트웨어다.

언어 모델은 사용자의 요청을 이해하고, 코드와 문서를 읽어 다음에 수행할 행동을 결정하는 생각 엔진 역할을 한다.
즉 어떤 파일을 찾아야 하는지, 어떤 코드를 수정해야 하는지, 어떤 테스트를 실행해야 하는지를 판단한다.

샌드박스는 에이전트가 작업하는 격리된 실행 환경이다.
에이전트는 이 안에서 파일을 읽고 쓰거나 명령어를 실행할 수 있지만, 권한이 제한되어 있기 때문에 프로젝트 바깥의 파일을 임의로 수정하거나 위험한 명령을 바로 실행하지 못하게 막을 수 있다.
이 덕분에 코딩 에이전트는 실제 개발 환경에 접근하면서도, 사용자가 허용한 범위 안에서만 행동하도록 제어된다.

[에이전트 제어 장치](#6-에이전트-제어-장치-agent-harness)는 언어 모델이 실제 개발 환경에서 코드를 읽고 편집하고 실행할 수 있도록 도와주는 소프트웨어다.
LLM이 "이 명령을 실행하겠다" 또는 "이 파일을 수정하겠다"고 판단하면, 에이전트 제어 장치가 그 요청을 받아 권한을 확인하고, 샌드박스 안에서 실행한 뒤, 결과를 다시 LLM에게 전달한다.
또한, 언어 모델의 맥락 공간(context window)이 가득 찬 경우 맥락 압축을 통해 에이전트가 장기 작업을 수행할 수 있도록 돕는다.
즉, 에이전트 제어 장치는 코딩 에이전트를 하나의 답변만 생성하는 챗봇이 아니라 실제 개발 작업을 반복 수행하는 시스템으로 만들어 주는 핵심 구성 요소이다.

## 4. 도구 호출

도구 호출은 코딩 에이전트가 실제 개발 환경에서 행동할 수 있게 해 주는 핵심 기능이다.
여기서 도구는 언어 모델 바깥에 있는 모든 소프트웨어를 가리킨다.
예를 들어 터미널 명령어, 테스트 실행기, 웹 검색, GitHub 이슈 조회 API가 모두 도구가 될 수 있다.
언어 모델은 자연어로 사용자의 요청을 이해한 뒤, 필요한 정보를 얻거나 작업을 수행하기 위해 어떤 도구를 어떤 인자로 호출할지 결정한다.

도구 호출(tool calling)은 보통 사람이 직접 보는 문장이 아니라, 언어 모델이 실행 환경에 보내는 구조화된 요청이다.
예를 들어 파일 검색 도구를 호출하려고 한다면 언어 모델이 다음과 같은 문자열을 스스로 생성한다.

```json
<TOOL>
{
  "tool": "run_shell_command",
  "arguments": {
    "command": "grep -R \"login\" ."
  }
}
<\TOOL>
```

위 예시는 `run_shell_command`라는 도구에 `grep -R "login" .` 명령을 실행해 달라고 요청하는 도구 호출을 의미한다.
도구 호출 패턴이 감지되면 [에이전트 제어 장치(Agent Harness)](#6-에이전트-제어-장치-agent-harness)가 셸에서 언어 모델이 실행하려고 하는 명령어를 대신 실행해 준 후 결과를 언어 모델에게 다시 입력으로 보낸다.

코딩 에이전트의 능력은 모델 자체의 코드 지식뿐 아니라, 도구를 적절한 순서로 사용하고 그 결과를 해석하는 능력에 크게 의존한다.
최근에는 여러 번의 도구 호출을 거쳐야 해결할 수 있는 문제를 대상으로 에이전트의 도구 사용 능력을 측정하는 벤치마크도 제안되고 있다<sup>[1](#tau_bench), [2](#terminal_bench)</sup>.

## 5. 작업 맥락의 이해

코딩 에이전트는 작업에 앞서 관련 파일을 검색하고, 디렉터리 구조를 살피고, 테스트와 설정 파일을 확인하면서 전체 흐름을 이해한다.
이는 기존 코드의 스타일과 설계를 해치지 않고 필요한 부분만 정확히 수정할 수 있도록 돕는다.

일반적인 탐색 순서는 다음과 같다.

1. `README.md`, 패키지 설정 파일, 빌드 설정 파일처럼 프로젝트의 목적과 실행 방법을 알려 주는 파일을 읽는다.
2. `src/`, `test/`, `docs/` 같은 주요 디렉터리를 살펴보며 코드가 기능별로 어떻게 나뉘어 있는지 확인한다.
3. 사용자의 요청과 관련된 키워드를 검색하여 실제로 수정해야 할 파일과 함수의 후보를 찾는다.
4. 테스트 파일이나 기존 사용 예시를 읽어, 코드가 어떤 동작을 기대하고 있는지 확인한다.

예를 들어 "회원 가입 시 이메일 검증을 추가해줘"라는 요청이 들어오면,
에이전트는 `signup`, `register`, `email`, `validation` 같은 단어를 `grep`을 사용하여 검색할 수 있다.
검색 결과에서 API 처리 코드, 입력 검증 코드, 테스트 코드를 찾고,
각 파일이 서로 어떤 순서로 호출되는지 읽으면서 수정 범위를 정한다.
이 과정은 사람이 처음 보는 프로젝트에 들어가서 코드를 파악하는 방식과 비슷하다.

`AGENTS.md`는 코딩 에이전트를 위한 특수한 지침 파일이다.
이 파일에는 코드 스타일, 테스트 실행 방법, 변경하면 안 되는 영역, 문서 작성 규칙처럼 에이전트가 따라야 할 규칙을 적을 수 있다.
예를 들어 "수정 후에는 반드시 `npm test`를 실행하라" 또는 "데이터베이스는 직접 수정하지 말라" 같은 지침이 들어갈 수 있다.

## 6. 에이전트 제어 장치(Agent Harness)

에이전트 제어 장치는 언어 모델과 개발 환경 사이를 이어주는 소프트웨어다.
언어 모델은 어떤 파일을 읽고, 어떤 명령을 실행하고, 어떤 코드를 수정할지 결정하지만, 파일 시스템이나 터미널을 직접 조작하지는 않는다.
에이전트 제어 장치는 언어 모델이 생성한 도구 호출을 받아 실제 환경에서 실행하고, 그 결과를 다시 언어 모델에게 전달한다.
이 반복 덕분에 코딩 에이전트는 단순한 코드 생성기가 아니라, 코드베이스를 읽고 수정하고 검증하는 작업 시스템으로 동작한다<sup>[4](#swe_agent)</sup>.

에이전트 제어 장치의 첫 번째 역할은 도구를 언어 모델 대신 실행해 주는 것이다.
코딩 에이전트가 사용할 수 있는 도구에는 파일 읽기, 파일 검색, 파일 수정, 셸 명령 실행, 테스트 실행, Git 명령, GitHub 이슈 조회 등이 있다.
언어 모델은 이 도구들을 직접 실행하는 것이 아니라, [4. 도구 호출](#4-도구-호출)의 예시와 같은 구조화된 요청을 만든다.
에이전트 제어 장치는 이 요청을 해석하여 실제 도구를 실행한다.

두 번째 역할은 작업 루프를 관리하는 것이다.
코딩 에이전트의 작업은 한 번의 답변으로 끝나지 않는다.
에이전트는 먼저 파일을 검색하고, 관련 코드를 읽고, 수정한 뒤, 테스트를 실행하고, 실패하면 다시 원인을 찾아 수정한다.
이 과정은 다음과 같은 순서로 반복된다.

1. 언어 모델이 다음 행동을 결정한다.
2. 에이전트 제어 장치가 그 행동이 허용되는지 확인한다.
3. 허용된 행동을 샌드박스 안에서 실행한다.
4. 실행 결과, 오류 메시지, 테스트 결과, 변경된 파일 정보를 수집한다.
5. 그 결과를 문자열로 바꾼 후 언어 모델에게 전달한다.
6. 언어 모델이 새 정보를 바탕으로 다음 행동을 결정한다.

세 번째 역할은 권한을 관리하는 것이다.
코딩 에이전트는 실제 파일을 수정하고 명령어를 실행할 수 있기 때문에 잘못 사용하면 위험하다.
예를 들어 모델이 실수로 중요한 파일을 삭제하거나, 프로젝트 바깥의 파일을 읽거나, 네트워크를 통해 민감한 정보를 외부로 보낼 수 있다.
에이전트 제어 장치는 작업 디렉터리, 파일 쓰기 권한, 네트워크 접근, 위험한 명령어 실행 여부를 제한해 준다.

네 번째 역할은 작업 맥락을 관리하는 것이다.
언어 모델의 맥락 공간에는 한 번에 넣을 수 있는 정보량에 한계가 있다.
큰 코드베이스 전체를 그대로 넣을 수 없기 때문에, 에이전트 제어 장치는 현재 작업에 필요한 파일 조각, 명령 실행 결과, 테스트 실패 로그, 이미 수정한 파일 목록 등을 선별해 모델에게 전달한다.
출력이 너무 길면 중요한 부분만 요약하고, 장시간 작업에서는 맥락을 압축하여 작업을 이어갈 수 있도록 도와준다<sup>[5](#claude_code)</sup>.

에이전트 제어 장치가 중요한 이유는 최근 코딩 에이전트의 성능이 모델 자체만으로 결정되지 않기 때문이다.
같은 언어 모델을 사용하더라도 어떤 도구를 제공하는지, 파일을 어떤 방식으로 보여 주는지,
편집을 어떻게 적용하는지, 테스트 결과를 어떻게 요약하는지에 따라 성능이 크게 달라질 수 있다.

## 7. 명세 주도 개발 (Spec-Driven Development)

명세 주도 개발은 무엇을 만들어야 하는지를 먼저 정리하고, 그 명세를 기준으로 설계, 구현, 테스트를 진행하는 개발 방식이다.

최근 코딩 에이전트를 사용하여 고품질 코드를 작성하기 위해 명세 주도 개발이 주목받고 있다.
코딩 에이전트는 자연어 요청만으로도 코드를 생성할 수 있지만, 요청이 모호하면 사용자가 의도하지 않은 코드를 만들 수 있다.
먼저 명세를 작성하고, 그 명세를 바탕으로 에이전트가 구현하도록 강제해 고품질의 코드를 생성하고, 코드를 명세와 비교하여 더 쉽게 검토하고 수정할 수 있다<sup>[6](#spec_kit), [7](#kiro_specs)</sup>.

명세 주도 개발의 일반적인 흐름은 다음과 같다.

1. **명세 작성**

   사용자가 만들 기능을 자연어로 설명한다.
   이때 단순히 "로그인 기능을 만들어줘"라고 쓰는 것이 아니라, 어떤 사용자가 어떤 입력을 넣고, 성공과 실패 상황에서 어떤 결과가 나와야 하는지까지 적는다.
   이 과정은 사용자가 전부 작성하기보다 언어 모델이 초안을 만들고 사람이 검토하는 방식으로 진행할 수 있다.

2. **설계 작성**

   요구 사항을 실제 코드 구조로 옮기기 전에, 어떤 파일과 모듈을 수정할지 정한다.

3. **작업 목록 생성**

   설계를 작은 구현 단위로 나눈다.
   예를 들어 "입력 검증 함수 추가", "로그인 API 수정", "실패 테스트 추가", "문서 업데이트"처럼 코딩 에이전트가 하나씩 수행할 수 있는 작업으로 나눈다.

4. **구현과 검증**

   코딩 에이전트가 작업 목록에 따라 코드를 수정하고 테스트를 실행한다.
   테스트가 실패하면 실패 로그를 다시 읽고, 명세와 구현이 어긋난 부분을 찾아 수정한다.
